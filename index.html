命令的分类

程序考虑 3 种类型的命令：

  1. 普通命令，即只调用某个程序的命令，如 ls, pwd 这种
  2. 重定向命令，如 ls >a.txt
  3. 管道命令，如 ls | wc

这三种命令可以组合成更复杂的命令，例如 cat < y | sort | wc > y1，但
是这条命令可以看作一个管道命令，管道左边为 cat <y，一个重定向命令；
管道右边为 sort | wc > y1，以此类推可以继续分解。

一些问题

管道相关

how does each command decide when to exit?

以 ls | wc 为例：

ls 将输出全部写入管道，exit。这时 ls 包含的所有 fd 也都关闭，因此
管道的 write end 被关闭。

wc 从管道中读取 ls 的输入，当 ls 结束，管道的 read end 变的不可
读，这时 wc 输出结果，exit。

what if reader didn’t close the write end?

测试发现，如果 reader 不在一开始将 p[1] 关闭的话，管道会一直处在可读
的状态，即使 writer 已经退出了。reader 将会一直试图读取，因此会无限阻
塞。

what if writer didn’t close the read end?

无影响。

how does the kernel know when to free the pipe buffer?

应当是管道 p 的 fd 的引用数都为 0 的时候？

fork 相关

what if sh didn’t fork for pcmd->left?

会导致 pcmd->right 无法执行。没有 reader 读取管道，pcmd->left 也会
阻塞，因此命令会一直等待，无法返回。

what if sh didn’t fork for pcmd->right?

在一些情况下，例如运行 ls | wc 时，这样做看起来没有什么影响。

但是在运行其他一些 writer 耗时较多的命令时，例如 sleep 10 | echo hi
时，命令会立即返回，而不等 sleep 结束。这样 shell 的行为与预期不符。
这是因为 echo 不需要输入，因此执行结束就返回了。而如果 reader 是需要
输入的，例如运行 sleep 10 | wc 时，程序会等待 10s 然后返回。

再来看一下运行 sleep 10 | echo hi 时发生的情况。由于 echo 不是
fork 出来的子进程，而是原来的进程直接 execv 出来的，因此它不会进行
wait 等待 sleep，而是直接退出。两个程序的状态变化：

  1. echo 进程打印输出，然后 exit，命令返回
  2. sleep 进程成为 orphand，成为 init 的子进程，10s 后 exit，init 执行 wait
